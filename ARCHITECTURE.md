## 一、产品定位（一句话）

> **hermes-dashboard 是面向AI原生开发者的统一指挥中心**——在一个可视化管理界面中管理所有隔离的Hermes项目Profile，每个Profile拥有独立记忆/SOUL/技能体系且互不干扰，同时所有产出数据自动汇聚到**盖娅之城统一数据库**，实现「项目隔离、数据统一」的一劳永逸开发体验。AI数智军团是其中的一个项目。

## 二、核心理念

### 2.1 为什么需要这个架构

当前状态：
```
chainke-dev (独立profile)  ── 独立记忆/SOUL/技能
gaia-city   (独立profile)  ── 独立记忆/SOUL/技能
zhairu      (独立profile)  ── 独立记忆/SOUL/技能
production  (独立profile)  ── 独立记忆/SOUL/技能
dev         (独立profile)  ── 独立记忆/SOUL/技能
  │
  └── ❌ 切换靠CLI · 数据不互通 · 无统一视图 · 开一堆窗口
```

目标状态：
```
      ┌────────────────────────────────────┐
      │      hermes-dashboard              │
      │    统一指挥中心（Web界面）           │
      └──────┬──────┬──────┬──────┬───────┘
             │      │      │      │
    ┌────────▼┐ ┌──▼───┐ ┌▼─────┐ ┌▼──────┐
    │链客宝   │ │盖娅  │ │数智  │ │招贤    │ 更多...
    │独立记忆 │ │独立  │ │军团  │ │独立    │
    │独立SOUL │ │记忆  │ │独立  │ │记忆    │
    │独立技能 │ │独立  │ │记忆  │ │独立    │
    └────┬────┘ └──┬───┘ └──┬──┘ └───┬───┘
         │         │        │         │

             ┌─────────▼─────────┐

             │  PostgreSQL 16    │
             │  :5435 chainke-pg │
             └───────────────────┘
```

### 2.2 核心设计原则

| 原则 | 含义 |
|:-----|:------|
| **项目隔离** | 每个Profile是独立Hermes进程，记忆/SOUL/技能完全隔离，互不污染 |
| **知识统一** | 所有Profile的产出数据单向同步到盖娅之城PG，形成统一知识资产 |
| **零侵入** | 不修改任何现有Profile、不替代记忆宫殿、不破坏已有基础设施 |
| **单向同步** | Profile→PG单向流动，删除PG不影响Profile数据完整性 |
| **增量构建** | 优先复用已有服务（盖娅:5057/链客:8003/PG:5435/看板:5003），不重复造轮 |

## 四、与现有体系的关系

### 4.1 与记忆宫殿（L0-L5）

| 层级 | 当前角色 | hermes-dashboard 中的角色 |
|:-----|:---------|:-------------------------|
| L0 前厅 | 3+X窗口/状态总览 | 保留为 Dashboard 首页数据源 |
| L1 图书馆 | 知识存储 | 跨Profile知识检索的索引源 |
| L2 档案馆 | 历史记录 | 保留，同步到PG作为归档加速 |
| L3 工作室 | 战略分析/蜂巢讨论 | 保留，Dashboard可引用分析结果 |
| L4 博物馆 | 作品展示 | 保留，Dashboard可展示 |
| L5 孵化室 | 产品开发 | 保留，各Profile的产出目标区 |

**原则**：记忆宫殿是**唯一权威源**，PG只是索引加速和跨项目检索的副本。

### 4.2 与盖娅之城

| 维度 | 盖娅之城 (Gaia City) | hermes-dashboard |
|:-----|:---------------------|:-----------------|
| 角色 | 呈现层 · 可视化指挥 | 工程运维层 · 开发管理 |
| 目标 | 展示军团状态/公民档案/数据洞察 | 管理Profile/开发流程/数据同步 |
| 用户 | 管理者/访客 | 开发者/指挥官 |
| 数据 | 读取盖娅PG + SSE实时推送 | 管理Profile生命周期 + 调度同步管道 |
| 关系 | 盖娅展示最终结果 | Dashboard管理开发过程 |

**两者互补，不重叠**：Dashboard管开发过程，盖娅管视觉呈现和军团指挥。

### 4.3 与AI数智军团

AI数智军团是 hermes-dashboard 中的一个 **Profile 项目实例**，但同时具备四重身份：

1. **自举项目** — 军团自身就是开发者之一（白泽指挥军团）
2. **能力提供者** — 军团的数字员工可响应 Dashboard 的调度
3. **监控对象** — Dashboard 监控军团各服务端口健康状态
4. **架构试金石** — 军团是最大规模的使用者，验证架构可扩展性

## 五、技术选型

### 5.1 前端 (Dashboard Web UI)

| 层级 | 选型 | 理由 |
|:-----|:------|:------|
| 框架 | React 19 + TypeScript | 生态成熟、组件化 |
| 构建 | Vite 6 | 极速HMR、Tree Shaking |
| UI | shadcn/ui + TailwindCSS 4 | 美观、可定制、轻量 |
| 状态管理 | Zustand | 比Redux轻10倍 |
| 实时通信 | Socket.IO (WebSocket fallback) | 日志流/状态变更推送 |
| 图表 | ECharts 5 | 丰富、SSE仪表盘适配 |

### 5.2 后端 (Control Plane)

| 层级 | 选型 | 理由 |
|:-----|:------|:------|
| 框架 | FastAPI (Python 3.12+) | 异步原生、Pydantic v2 |
| ASGI | Uvicorn + Gunicorn | 生产级Worker管理 |
| ORM | SQLAlchemy 2.0 (异步) + Alembic | 成熟、迁移工具 |

### 5.3 Profile集成方式

| Profile操作 | 实现方式 |
|:-----------|:---------|
| 创建 | `subprocess.Popen(['hermes', 'profile', 'create', name])` |
| 启动 | `subprocess.Popen(['hermes', '-p', name, 'chat'])` + WebSocket捕获stdout |
| 停止 | 发`/exit`命令 → wait PID |
| 状态 | 检测PID存活 + socket端口扫描 |
| 命令注入 | 通过stdin写入 → stdout/stderr流捕获 → WebSocket推前端 |

## 七、关键设计决策

| 决策项 | 选择 | 被否决的方案 | 理由 |
|:-------|:-----|:------------|:------|
| 前端框架 | React 19 | Vue/Preact/Lit | 生态最成熟 |
| 后端框架 | FastAPI | Flask/Django/NestJS | 异步原生+类型安全 |
| 实时通信 | WebSocket/SSE | gRPC Stream | 浏览器兼容性更好 |
| 数据同步方向 | 单向Profile→PG | 双向同步 | 防止Profile被污染 |
| Profile集成 | subprocess+stdin管道 | Hermes SDK/Python API | 零侵入，不修改Hermes |
| 前端部署 | 独立Docker | 嵌入盖娅Flask | 不耦合盖娅部署 |

## 八、风险与缓解

| 风险 | 等级 | 缓解方案 |
|:-----|:----:|:---------|
| Profile进程意外退出 | 🔴 | 自动重启 + 心跳检测 + 日志保留 |
| 数据同步冲突 | 🟡 | 时间戳版本控制 + 人工裁决入口 |
| 性能瓶颈（多Profile同时运行） | 🟡 | 资源配额 + 单个Profile限制 + 懒加载 |
| 跨Profile命令污染 | 🔴 | 严格进程隔离 + stdin白名单 + 沙箱 |
| .env/API Key泄露 | 🔴 | Web UI不显示密钥 + 仅从文件读取 |

## 十、总结

hermes-dashboard 不是替代品，而是**倍增器**：

- **不替代** 记忆宫殿（仍为唯一权威源）
- **不替代** 盖娅之城（盖娅做呈现层，Dashboard做工程层）
- **不替代** 各Profile独立运行（保持隔离性）
- **不替代** Hermes CLI（仍可独立使用）

而是将所有这些**统一在一个界面**，让开发者：
- 打开一个网页 = 管理所有项目
- 每个项目独立运转 = 互不干扰
- 所有数据汇聚 = 一劳永逸的知识沉淀
